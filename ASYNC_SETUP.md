# Async RegIntel AI Backend Setup Guide

This guide covers the setup and configuration of the refactored RegIntel AI backend with async task processing and multi-model fallbacks.

## Overview

The refactored backend includes:
- **Celery + Redis** for asynchronous task processing
- **ModelRouter** with Ollama primary and OpenAI fallback
- **SQLAlchemy** database with DecisionLog table
- **Task polling** support in the frontend

## Prerequisites

1. **Redis Server**: Install and start Redis
2. **Ollama**: Install and run Ollama with Llama 3.1:8b model
3. **OpenAI API Key**: Optional, for fallback model
4. **Python Dependencies**: Install from requirements.txt

## Environment Configuration

Update your `.env` file with the following variables:

```env
# Existing configuration
OLLAMA_BASE_URL=http://localhost:11434
DATABASE_URL=sqlite:///data/decision_logs.db

# New Redis and Celery configuration
REDIS_URL=redis://localhost:6379/0
OPENAI_API_KEY=your_actual_openai_api_key_here
CELERY_BROKER_URL=redis://localhost:6379/0
CELERY_RESULT_BACKEND=redis://localhost:6379/0
```

## Setup Instructions

### 1. Install Dependencies

```bash
pip install -r requirements.txt
```

### 2. Start Redis Server

```bash
# Windows (if using Redis for Windows)
redis-server

# Linux/MacOS
redis-server
```

### 3. Start Ollama and Download Model

```bash
# Start Ollama
ollama serve

# Download Llama 3.1:8b model
ollama pull llama3.1:8b
```

### 4. Start Celery Worker

Open a new terminal and run:

```bash
python start_celery_worker.py
```

### 5. Start FastAPI Server

Open another terminal and run:

```bash
uvicorn src.main:app --reload --host 0.0.0.0 --port 8000
```

## Frontend Integration

### Using the Updated apiService

```typescript
import { apiService } from '../services/apiService';

// Trigger async ingestion
const taskResponse = await apiService.triggerIngestion(['source1', 'source2']);
const taskRunner = apiService.createTaskRunner(taskResponse.task_id);

// Poll for completion with progress updates
try {
  const result = await taskRunner.wait((status) => {
    console.log(`Progress: ${status.progress}% - ${status.status}`);
  });
  console.log('Task completed:', result);
} catch (error) {
  console.error('Task failed:', error);
}

// Query with model routing
const queryResult = await apiService.queryRegulation({
  query: "Latest RBI guidelines"
});
console.log(`Model used: ${queryResult.model_used}`);
console.log(`Fallback triggered: ${queryResult.fallback_triggered}`);
```

## Model Router Behavior

### Primary Model (Ollama)
- **Model**: Llama 3.1:8b
- **Timeout**: 10 seconds
- **Confidence Threshold**: 70%

### Fallback Logic
The ModelRouter triggers fallback when:
1. Primary model takes > 10 seconds
2. Confidence score < 70%
3. Primary model fails or is unavailable

### Fallback Model (OpenAI)
- **Model**: GPT-4o-mini
- **Timeout**: 30 seconds
- **Requires**: Valid OpenAI API key

## Database Schema

### DecisionLog Table
- `id`: Primary key
- `timestamp`: Decision timestamp
- `agent_name`: Agent that made the decision
- `input_data`: Input provided to agent
- `agent_output`: Output generated by agent
- `confidence_score`: AI confidence score (0-100)
- `model_used`: Model used (ollama/openai)
- `fallback_triggered`: Whether fallback was used
- `processing_time`: Time taken in seconds

## Troubleshooting

### Common Issues

1. **Redis Connection Failed**
   - Ensure Redis server is running
   - Check REDIS_URL in .env file

2. **Celery Worker Not Starting**
   - Verify Redis connection
   - Check for import errors in tasks.py

3. **Ollama Model Not Available**
   - Ensure Ollama is running: `ollama serve`
   - Download model: `ollama pull llama3.1:8b`

4. **OpenAI Fallback Not Working**
   - Verify OPENAI_API_KEY in .env
   - Check API key validity and credits

5. **Database Connection Issues**
   - Ensure data directory exists
   - Check DATABASE_URL path

### Logs and Debugging

- **FastAPI logs**: Check terminal running uvicorn
- **Celery logs**: Check terminal running celery worker
- **Database logs**: Check SQLAlchemy logging output

## Performance Considerations

- **Celery Workers**: Adjust concurrency based on CPU cores
- **Redis Memory**: Monitor Redis memory usage
- **Database**: Consider PostgreSQL for production
- **Model Caching**: Implement response caching for repeated queries

## Production Deployment

For production deployment:
1. Use Redis Cluster for high availability
2. Deploy Celery workers across multiple machines
3. Use PostgreSQL instead of SQLite
4. Implement proper logging and monitoring
5. Set up task queues with priorities
6. Configure proper error handling and retries
